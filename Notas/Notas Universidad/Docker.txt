        max_attempts: 4
        window: 120s

Esquema optimo de despliegue - Comportamiento de Traefkik - Respiesta SIU

No hay un esquema óptimo de cómo escalar la instalación, además, va a variar mucho dependiendo de cada caso. Lo que podemos hacer es darte una serie de recomendaciones y puntos a tener en cuenta.

1) Escalado horizontal:
Inicialmente, tenés las recomendaciones de la web de expedientes:
https://expedientes.siu.edu.ar/docs/rendimiento/#escalabilidad-horizontal
Acá tenés que tener en cuenta que los servicios que vayas a escalar tienen que poder ser escalables, y las condiciones de esa escalibilidad. Por ejemplo, los distintos servicios de Araí Usuarios:
https://expedientes.siu.edu.ar/docs/rendimiento/#arai-usuarios
Además, también tener en cuenta de acuerdo a tu instalación, cuáles servicios es necesario que tengan replicas y cuáles no. Quizás no es necesario iniciar con todos los servicios replicados.

2) Escalado vertical:
Esto ya se refiere a los recursos que vas a asignar a cada servicio.
Cada servicio tiene un mínimo y un máximo de recursos asignados (memoria y procesador):
resources:
   limits:
       cpus: '0.2'
       memory: 70M
   reservations:
       cpus: '0.05'
       memory: 64M
Los limits indican el máximo de recursos que va a usar el servicio, y los reservations indican el mínimo de recursos.
En este punto tenés que considerar el servicio del api server de SUDOCU. Este servicio en limits necesita al menos 1 CPU y 4096M de RAM.
Otros servicios que también necesitan extender el limit son el stamper de Araí Docs y pdf de SUDOCU.
Pueden usar el comando docker stats para ir viendo los recursos que está utilizando cada servicio.

3) Sobre Traefik:
El proxy reverso de Traefik corre exclusivamente sobre nodos master, esto es fundamental.
Si tienen expuesto solo nodo para que el tráfico ingrese, entonces Traefik debe correr en este nodo. Si el servicio de Traefik está replicado en todos nodos master, tienen que tener esto en cuenta y exponer los 3 nodos y configurar sus DNS adecuandamente para que tenga esto en cuenta.
Hay algunos enlaces con información sobre este tema de Traefik, HA y DNS, pero sería ideal que lo analicen con su área de IT.
Como mínimo hay que tener en cuenta estos puntos al momento de hacer el despliegue de toda la plataforma del SEEI, por todo esto es que les recomendamos iniciar su despliegue sobre un nodo máster (el mismo en el que corre el proxy reverso de traefik). Una vez que verifiquen que todas las operaciones funcionan correctamente comenzaría con el escalado de los servicios, tanto horizontal como vertical. Y a medida que van escalando, van verificando si los distintos errores que tienen persisten o se van resolviendo. Pero es un proceso que lo llevaría más paso a paso.
Estas recomendaciones son las mismas que charlamos durante la última reunión, también consideramos que es la mejor forma de detectar en qué momento empiezan a aparecer los errores.

Para añadir imagenes al docker:

1- Se debe buscar el nombre de la imagen jpg a cambiar en la pagina y buscarla en el docker

2- Copiarla al docker
docker cp   img_meta.png   huarpe_webapp.1.xipsjnpjw8qmde8wcudl5hwjg:/usr/local/app/web/img/logos/

3- Ingresar al docker
docker exec -it usuarios_api.1.pjdsdoptyae8fw5iw5rxfigk2 /bin/bash

4- Buscar en el docker 
find . -iname *.png

5- Renombrar y reemplazar la imagen:
mv logo-blanco-y-negro.png logo-blanco-y-negro.png.bk
mv img_meta.png logo-blanco-y-negro.png

Tener en cuenta que si el ID del docker no esta en el nodo master, puede estar en un nodo Worker. Por lo que este proceso se debe correr en todos los nodos.

Limites de memoria, memory limit:

Memory reservation (MB)  	Minimum memory available on a node to run a task (set to 0 for unlimited) 

Memory limit (MB)  	Maximum memory usage per task (set to 0 for unlimited) 

Asi quedaria la pestaña Deploy con las modificaciones para que NO tenga maximo de memoriy usage:

    deploy:
      replicas: 1
      resources:
#        limits:
#          cpus: '1'
#          memory: 1024M
        reservations:
          cpus: '0.2'
          memory: 256M

Recordar que el numero de replicas de los contenedores tiene que ser consistente con la cantidad de nodos, asi se puede producir una replica por nodo
